インタフェース（第13回）

この単元の前に：抽象クラスの復習

前回は「抽象クラス」で、サブクラスに「必ずこのメソッドを実装してね」と任せるやり方を学びました。短くおさらいします。

復習：前回までに学んだこと

  • 抽象クラス — abstract class で宣言。1つでも抽象メソッドがあると、そのクラスは new できない。
  • 抽象メソッド — 中身を書かず、サブクラスで「必ず実装する」ことを強制する。サブクラスでは extends で1つだけ継承できる。
  • 実装したメソッド — オーバーライドするときは public をつけておく（省略しても暗に public になる）。

この単元で学ぶこと

この単元では「インタフェース」を学びます。抽象クラスを極限まで突き詰めた「メソッドの約束だけ」を表す仕組みで、次回の多態性の土台になります。

【1】具体的な学習目標

この単元を終えたとき、以下のことができればゴールです。

  • インタフェースの役割（「このメソッドを必ず実装する」という約束だけを定義する）を説明できる。
  • interface と implements の書き方を理解する。
  • 「クラスは1つしか継承できないが、インタフェースは複数実装できる」ことを理解する。

【2】導入：メソッドの約束だけを切り出したものがインタフェース

カバ先生：「前回は抽象クラスで、サブクラスに『必ずこのメソッドを実装してね』と任せましたね。それをさらに進めると、中身がまったくないメソッドの『形』だけが並んだものになります。Java では、それをインタフェースとして、クラスとは別に定義します。」

ユウタ：「インタフェースには、どんなメソッドを書けばいいか、名前と引数と戻り値の約束だけを書く、という理解でいいですか？」

カバ先生：「その通りです。そして大事な違いがもう一つ。クラスは extends で1つしか継承できませんが、インタフェースは implements で複数実装できます。だから『移動できる（Movable）』と『呼び込みできる（Speakable）』を別々のインタフェースにして、1つの移動販売機クラスが両方実装する、といったことができます。」

ナナコ：「実装するとき、そのメソッドには public をつける、と前回習いました。インタフェースで決めたメソッドをクラスで実装するときも、同じように public でオーバーライドする、という理解でよいでしょうか。」

カバ先生：「はい。インタフェースのメソッドは省略すると public abstract として扱われ、実装する側でも public を省略すると暗に付きます。読みやすさのため、実装側には public を明示しておくといいですよ。」

既存クラスに新たな価値観を与えるためのベースとして利用するのがインタフェースです。

比較：継承とインタフェース（使い分けの整理）

観点 継承（extends） インタフェース（implements）
目的 同じ種類のクラスに新たなバリエーションを付与して、サブクラスを作成する。 独立した別の機能をあたえて、そのメソッドの実装を強制するためのもの。
関係のイメージ 「〜は〜の一種（is-a）」を表す。 「〜は〜ができる（can-do）」を表す。
数 クラスの継承は 1つだけ（単一継承）。 インタフェースは複数実装できる（多重実装）。

【3】インタフェース：メソッドの形だけを定義する

インタフェースは、メソッドの「名前・引数・戻り値」の約束だけを書きます。中身（本体）は書きません。クラスがそのインタフェースを implements すると、約束したメソッドをすべて実装する必要があります。

  • interface 名前 — インタフェースの宣言。中にはメソッドの形（シグネチャ）だけを書く。
  • implements インタフェース名 — クラスが「このインタフェースで決めたメソッドを実装します」と宣言する。複数書ける（implements A, B, C）。
  • インタフェースのメソッドは修飾子を省略すると public abstract として扱われる。実装側では @Override を付け、public で実装する。

インタフェースの作成と多重実装

Speakable.java は void speak(); だけを持たせます。移動販売機は「移動ユニット」と「音響ユニット」という2つの異なる能力（インタフェース）を装備するイメージです。

図：1つのクラスが Movable と Speakable の2つを実装（多重実装）

※図の差し込みエリア（LearningBoxで手差し）
// 多重実装：class VendingMachine implements Movable, Speakable
public interface Movable { void move(); }
public interface Speakable { void speak(); }

public class VendingMachine implements Movable, Speakable {
    private String name;
    public VendingMachine(String name) { this.name = name; }
    @Override public void move() { System.out.println(name + "が指定の場所へ移動しました"); }
    @Override public void speak() { System.out.println("ピロリロ〜ン♪ 淹れたてのコーヒーはいかがですか？"); }
}

このように implements Movable, Speakable と複数カンマで並べると、1つのクラスが2つのインタフェースのメソッドをすべて実装する必要があります。次回の多態性では、Movable 型や Speakable 型の変数で受け、同じ move()・speak() を呼ぶ扱いを学びます。

【5】練習問題（VendingMachine で営業フローを実装）

【問題】Main クラスで「移動 → 呼び込み」の営業フローを実装する
上記の VendingMachine（implements Movable, Speakable）を使い、指定の場所に移動（move）したあと、元気よく呼び込み（speak）をするという一連の営業フローを Main クラスの main メソッドで実装してください。
▶ ヒント

VendingMachine のインスタンスを作り、まず move() を呼んでから speak() を呼ぶ順番にします。例：VendingMachine vm = new VendingMachine("コーヒー号"); vm.move(); vm.speak();

▶ 解答例（Main の main 内）

【解説・確認用】移動してから呼び込みの順で実行します。

VendingMachine vm = new VendingMachine("コーヒー号");
vm.move();
vm.speak();

【6】まとめ

  • インタフェース — メソッドの「形」だけを定義する。中身は書かず、実装するクラスに任せる。
  • implements で複数のインタフェースを実装できる。クラスの extends は1つだけ。
  • 実装したメソッドには public をつけておく。次回は多態性（同じ型で受け、実際のインスタンスに応じて振る舞いが変わる）を学びます。

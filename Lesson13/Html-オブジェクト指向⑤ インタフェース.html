インタフェース（第13回）

この単元の前に：抽象クラスの復習

前回は「抽象クラス」を学びました。スーパークラスを基本とするだけにし、抽象メソッドでサブクラスに実装を任せる仕組みです。

この単元で学ぶこと

スーパークラスを極限まで抽象化すると、抽象メソッドだけのクラスができあがります。それはインタフェースとして別に定義されます。次回の多態性の土台になります。

【1】具体的な学習目標

  • インタフェースの役割（抽象メソッドのみを約束として定義する）を説明できる。
  • interface と implements（class定義側のキーワード）の書き方を理解する。
  • 単一継承ではなく、インタフェースは複数実装可能であることを理解する。
  • 実装したメソッドには public をつけておく。

【2】導入：極限まで抽象化するとインタフェース

カバ先生：「スーパークラスを考えて、極限まで抽象化していけば、抽象メソッドだけのクラスができあがる。それはもはや抽象メソッドのみになるので、インタフェースという別のものとして定義するんだ。」

ユウタ：「抽象メソッドのみだから、abstract の記述も不要になるんですね。」

ナナコ：「インタフェースは複数実装できるから、移動型自販売機に move を足す、みたいな拡張ができますね。」

【3】インタフェース：メソッドの約束

インタフェースは、メソッドの「名前・引数・戻り値」だけを決めた約束です。中身は書きません。クラスは implements インタフェース名 で、その約束を実装します。

  • interface 名前 — メソッドの形だけを定義。本体は ; で終える。abstract は不要。
  • class クラス名 implements インタフェース名 — クラス定義側のキーワードは implements。複数のインタフェースをカンマで並べられる。
  • 実装したメソッドには public をつけておく。

【図の挿入場所】インタフェースの説明用（実装の関係・複数 implements など）

例：移動型自動販売機（複数実装）

既存の自販機に「動く」という役割を足したい場合、Movable のようなインタフェースを別に定義し、implements VendMachine, Movable のように複数実装できます。

// インタフェース：メソッドの形だけを決める
public interface CanSell {
    void buy();
}

// クラスが複数実装する例（extends 1つ + implements 複数）
public class Vending2Item extends VendMachine implements CanSell {
    // 既に buy() を持っているので、CanSell の約束を満たしている
}

【4】抽象クラスとインタフェースの比較

項目 抽象クラス インタフェース
キーワード（class定義側） extends implements
継承・実装の数 1つだけ（単一継承） 複数可能
抽象メソッド 1つ以上あると抽象クラス メソッドは原則すべて抽象（約束だけ）
具象メソッド 持てる 持たない（※Java 8 以降は default で持てるが、入門では扱わない）
フィールド 持てる 定数（public static final）のみ
使い分けの目安 共通の土台（実装）があり、is-a の関係がはっきりしている 振る舞いの約束だけほしい、複数の役割を足したい

【5】まとめと次回へ

  • インタフェース — 抽象メソッドのみを約束として定義。class定義側では implements で実装。複数実装可能。
  • 実装したメソッドには public をつけておく。
  • 次回はポリモーフィズム（多態性）で、抽象クラスとインタフェースの両方を型として使い、同じメッセージで異なる振る舞いを扱います。

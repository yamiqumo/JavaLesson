# Lesson12〜14：抽象クラス・インタフェース・ポリモーフィズムの流れ

## 共通で伝えること（抽象クラス・インタフェースどちらにも）

- **オーバーライド／実装したメソッドには `public` をつけておく。**  
  （注：Java の仕様では、これらは省略しても暗に `public` が付いたものとして扱われます。ただし読みやすさのため、明示的に `public` を書くようにしておくとよいです。）

---

## 抽象クラス（Lesson12）

- サブクラスで書き換えるのなら、そもそもスーパークラスに具体的に書く必要はない。
- なので、**基本とするためのクラスとしてしか利用しない**なら、スーパークラスのメソッドを**抽象メソッド**にします。
- **1つでも抽象メソッドがあるクラス**を**抽象クラス**といいます。

## インタフェース（Lesson13）

- スーパークラスを考えて、**極限まで抽象化していけば**、抽象メソッドだけのクラスができあがる。
- それはもはや**抽象メソッドのみ**になるので、**インタフェース**という別のものとして定義されます。
- 抽象メソッドのみなので、`abstract` の記述ももはや不要になります。
- **単一継承ではなく、インタフェースは複数の継承（実装）が可能**になります。  
  例：移動型自動販売機にする場合に、`move` という別の要素を追加したい → 別のインタフェース（例：`Movable`）を `implements` することで「動く」という役割を足す、といった形で拡張できます。

### 抽象クラスとインタフェースの比較

| 項目 | 抽象クラス | インタフェース |
| :--- | :--- | :--- |
| キーワード（class定義側） | `extends` | `implements` |
| 継承・実装の数 | 1つだけ（単一継承） | 複数可能 |
| 抽象メソッド | 1つ以上あると抽象クラス | メソッドは原則すべて抽象（約束だけ） |
| 具象メソッド | 持てる | 持たない（※Java 8 以降は `default` で持てるが、入門では扱わない） |
| フィールド | 持てる | 定数（`public static final`）のみ |
| 使い分けの目安 | 共通の土台（実装）があり、is-a の関係がはっきりしている | 振る舞いの約束だけほしい、複数の役割を足したい |

## ポリモーフィズム（Lesson14）・まとめ

## 講義・テキストで押さえておくとよいこと

1. **用語の対応**
   - 抽象クラス → `extends`（1つだけ）
   - インタフェース → `implements`（複数可）
   - 「継承」vs「実装」の言い分けを一度説明しておくと、後で「クラスは1つ、インタフェースは複数」の理由が伝わりやすいです。

2. **インタフェースのメソッドは暗に `public abstract`**
   - インタフェース内のメソッドは、修飾子を省略すると `public abstract` として扱われます。実装クラス側では、そのメソッドをオーバーライド（実装）するときに、先述のとおり暗に `public` が付きます。

3. **いつ抽象クラスでいつインタフェースか**
   - 「共通のフィールドやメソッドの実装がある」「is-a の関係がはっきりしている」→ 抽象クラス。
   - 「振る舞いの約束だけほしい」「複数の役割を足したい」→ インタフェース。  
   移動型自販機の例（既存の型 + `Movable`）は、インタフェースの「複数実装」の動機づけとしてそのまま使えます。

4. **Java 8 以降の `default` メソッド**
   - インタフェースに `default` でデフォルト実装を持たせられる話は、入門では「後から知ればよい」程度にし、Lesson13 では「インタフェース＝メソッドの約束だけ」で通すとシンプルです。

5. **ポリモーフィズムの具体例**
   - 抽象クラス型・インタフェース型の変数に、具象クラスのインスタンスを入れてメソッドを呼ぶと、実際にはオーバーライド／実装した側が動く、という流れを 1〜2 個のコード例で示すと、Lesson14 のまとめになります。

以上を、Lesson12（抽象クラスのみ）→ Lesson13（インタフェース）→ Lesson14（ポリモーフィズム）の順で使う想定です。
